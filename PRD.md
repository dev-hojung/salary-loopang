## 0. Executive Summary (핵심 요약)

### 0.1 한 줄 정의
**딴짓메이트**는 *"덜 일할수록 이기는"* 역설을 코어 메카닉으로 삼은, **로그인 없는 실시간 멀티플레이 직장인 밈 놀이**다. 진짜 상품은 게임이 아니라 **"동료를 초대할 핑계"**이며, 초대가 곧 유머의 완성이자 성장 엔진이다.

### 0.2 현재 위치
- 코어 플레이 루프가 구현된 **MVP 웹앱 완성 단계** (Next.js 16 · React 19 · Supabase Realtime · Railway 배포).
- 실시간 급여 카운터, 생산성 랭킹(낮을수록 상위), 부장님 경보, 1:1 대결, 미니게임 3종, 칭호·업적, 명예의 전당, 일별 루팡왕, 칼퇴 세리머니까지 **재미의 뼈대는 이미 갖춰져 있다.**
- 도메인 순수 로직(`lib/salary.ts`, `lib/achievements.ts`)은 테스트까지 분리되어 **코드 품질의 강점**이다.

### 0.3 런칭을 가로막는 단 하나의 근본 이슈
> **신뢰 경계(trust boundary)의 부재.** RLS가 전면 개방(`using(true) with check(true)`)되어 있고 인증이 없으며, 게임 결과를 클라이언트가 직접 DB에 쓴다. 그 결과 **누구나 anon 키로 남의 방 데이터를 읽고·조작**할 수 있다 — 랭킹 위조, 방 그리핑(부장님 경보 강제 토글), 전체 방/닉네임 덤프가 모두 가능하다.

이 하나가 개발 파트의 **Critical 런칭 블로커**이며, 동시에 앱 전환(푸시 알림)의 선행 조건이기도 하다. **P0의 본질은 "모든 mutation을 서버 경유 + 최소 신원 검증"으로 신뢰 경계를 세우는 것**이고, 이 작업 하나가 대부분의 보안·어뷰징 이슈를 동시에 닫는다.

### 0.4 4개 도메인 요약

| 도메인 | 핵심 결론 | 런칭 관문(P0) |
|---|---|---|
| **① 기획** | 신기능이 아니라 **리텐션·바이럴 루프를 완성**하는 것이 런칭. NSM = **주간 함께 논 방-세션 수**. | 초대/공유 루프 강화, 빈 방 대응, 어제의 루팡왕 데일리 훅, 공유 결과 카드, 기본 계측 |
| **② 개발** | **RLS 전면 개방 + 인증 부재 + 클라이언트 권위 write**가 단일 Critical 블로커. | 모든 mutation 서버 경유 + 최소 신원, 레이트리밋, 관측성, 헬스체크, CI 게이트 |
| **③ 마케팅** | **오가닉 바이럴이 곧 유통.** K-factor ≥ 1 넘기 전엔 유료 0원. | 공유 트리거(루팡왕/칭호 이미지 카드), 커뮤니티 시딩, 숏폼 "부장님 경보" 상황극 |
| **④ 앱 전환** | 런칭에 네이티브 앱 불필요. **PWA로 시작**, 진짜 선행 과제는 **서버 사이드 알림 트리거**. | (앱은 런칭 게이트 아님) — 단, 푸시의 전제인 백엔드 트리거는 개발 P1과 연동 |

### 0.5 가장 중요한 교차 통찰 (Cross-cutting Insight)
런칭 성패를 가르는 축은 결국 **두 갈래**이며, 네 도메인이 이 둘로 수렴한다:

1. **신뢰할 수 있는 서버 백엔드** — (개발 P0의 서버 권위 전환) = (앱 전환 Phase 0의 알림 트리거 백엔드). **같은 뿌리의 작업**이다. 이걸 하면 랭킹 신뢰성·어뷰징 방어·재참여 푸시가 한꺼번에 열린다.
2. **자랑스러운 공유 루프** — (기획 P0의 공유 결과 카드/데일리 훅) = (마케팅의 공유 트리거 요청). **같은 기능 세트**다. 이걸 하면 K-factor가 오르고 오가닉 성장이 붙는다.

→ **런칭 팀은 이 두 트랙에 집중하면 된다.** 나머지는 P1 이후로 미뤄도 안전하다.

## 1. 기획 (Product)

> **제품 원칙(Tone Charter):** "모든 지표는 아무 의미가 없습니다." 딴짓메이트는 성과를 측정하는 도구가 아니라, 성과 측정을 **패러디**하는 놀이다. 모든 기능 결정은 이 톤을 지키는 선에서만 이뤄진다. 실제로 업무를 방해하거나 죄책감을 유발하면 실패한 기능이다.

### 1.1 비전 & 문제 정의

**비전 한 줄**
> "직장인의 지루한 오후를, 아무 의미 없지만 다 같이 웃는 3분짜리 딴짓 놀이터로."

**우리가 건드리는 감정/니즈**

| 감정/상황 | 현재 직장인의 현실 | 딴짓메이트의 개입 |
|---|---|---|
| 오후의 무력감 | 2~4시, 집중은 안 되는데 퇴근은 멀다 | "지금 딴짓하는 사람 나뿐이 아니다"는 **동질감** |
| 익명적 유대 | 회사 익명 커뮤니티(블라인드)는 스트레스 배출 중심 | 스트레스가 아닌 **가벼운 공동 장난** |
| 무해한 반항 | 실제 태업은 리스크, 대놓고 놀 순 없음 | 리스크 0의 **역할극(루팡왕 놀이)** |
| 잠깐의 소속감 | 재택/분산 근무로 사무실 잡담 소멸 | 방 단위로 뭉치는 **가벼운 팀 놀이** |

**핵심 문제 정의**
- 직장인에게는 "3~5분간 죄책감 없이 딴짓할 명분"과 "혼자가 아니라는 감각"이 동시에 부족하다.
- 기존 킬링타임 앱(모바일 게임, 숏폼)은 **혼자** 소비하며, 직장 맥락과 무관해 공유·유대가 없다.
- 딴짓메이트는 "생산성이 **낮을수록** 이기는" 역설적 랭킹으로 죄책감을 웃음으로 전환하고, 방 코드 하나로 동료를 즉시 끌어들여 **집단 딴짓**을 놀이화한다.

**핵심 인사이트:** 이 제품의 진짜 상품은 "게임"이 아니라 **"동료를 초대할 핑계"**다. 초대가 곧 유머의 완성이다.

### 1.2 타깃 페르소나 & 세그먼트

**1차 타깃 (Primary) — "오후 2시의 김대리"**
- 25~35세, 사무직/IT/스타트업, 데스크톱 앞 근무 시간이 긴 직군
- 카톡·슬랙·블라인드 상시 사용, 밈/드립 문화에 익숙
- 사용 맥락: **점심 직후(13~14시)**, **애매한 오후(15~16시)**, **퇴근 직전(17~18시)** — 하루 3개의 "집중 저하 골든타임"
- 니즈: "잠깐 놀고 싶은데 혼자는 심심하고, 티는 안 나야 함"

**2차 타깃 (Secondary)**

| 세그먼트 | 특징 | 진입 경로 |
|---|---|---|
| 소규모 팀 리더/막내 | 팀 분위기 환기용으로 방을 개설 | 1차 타깃의 초대 → 방장화 |
| 재택/원격 근무자 | 사무실 잡담이 사라져 유대 갈증 | SNS/커뮤니티 바이럴 |
| 대학생/취준생 | 스터디·조모임 중 딴짓 | 초대 링크 확산 |
| 밈 소비층(비직장인) | "웃긴 웹서비스" 자체로 유입 | 캡처/공유 바이럴 |

**안티 페르소나 (의도적 비타깃)**
- 실제 생산성 관리/집중 습관 형성을 원하는 사용자 → 우리는 그 반대를 판다.
- 진지한 경쟁·랭킹 몰입 게이머 → 우리 지표는 "아무 의미 없음"이 컨셉.

**사용 맥락 요약:** 세션은 **짧고(3~10분), 반복적(하루 2~3회), 근무 시간대 집중**. 모바일보다 **데스크톱 브라우저 탭**이 1차 환경(회사 PC).

### 1.3 핵심 가치제안 & 차별점

**가치제안(Value Proposition)**
> "혼자 딴짓하면 죄책감, 다 같이 딴짓하면 놀이. 로그인도 설치도 없이, 방 코드 4자리로 3초 만에 동료와 함께 월급루팡을 시작한다."

**차별점 매트릭스**

| 비교군 | 그들의 방식 | 딴짓메이트의 차별점 |
|---|---|---|
| 집중/뽀모도로 앱(Forest 등) | 생산성 **향상** 보상 | 생산성 **하락**을 보상하는 역설 유머 |
| 익명 직장 커뮤니티(블라인드) | 텍스트 기반 배출·정보 | **실시간 함께 하는 행위(놀이)** 기반 |
| 캐주얼 웹게임(Gartic 등) | 게임 자체가 목적, 맥락 없음 | **직장 맥락 밈** + 초저마찰 입장 |
| 팀 아이스브레이커 도구 | B2B, 세팅 무겁고 진지 | **로그인 0, 세팅 0**, 그냥 웃긴 것 |

**방어 가능한 차별점(Moat 후보)**
1. **초저마찰 입장:** 로그인·앱 설치 없이 닉네임+방코드. "지금 당장 다 같이" 실현.
2. **역설적 랭킹 IP:** "루팡왕/부장님 경보/칼퇴 세리머니" 등 직장 밈을 게임 메카닉으로 번역한 세계관.
3. **집단 실시간성:** 부장님 경보(방 전원 동시 위장) 같은 **함께여야만 재밌는** 순간들 → 혼자 쓰는 앱이 복제하기 어려움.

**한 문장 포지셔닝:** "딴짓메이트는 직장인을 위한 *반(反)생산성* 실시간 멀티플레이 밈 놀이다."

### 1.4 북극성 지표(North Star)와 핵심 KPI

**North Star Metric (NSM)**
> **주간 "함께 논 방-세션(Co-play Sessions)" 수**
> = 2명 이상이 동시에 접속해 3분 이상 지속된 방 세션의 주간 합계

- 선정 이유: 이 제품의 가치는 "혼자 접속"이 아니라 **함께 논 순간**에서 발생한다. DAU가 아닌 "함께함의 밀도"를 추적해야 톤과 사업이 어긋나지 않는다.

**KPI 트리**

| 레이어 | 지표 | 정의 / 목표(런칭 후 4주 벤치마크 가설) |
|---|---|---|
| **획득(Acquisition)** | 신규 방문자, 방 생성 수 | 랜딩→방 생성 전환율 ≥ 25% |
| **활성(Activation)** | "함께함 첫 경험"율 | 첫 세션에서 2명+ 동시 접속 도달 비율 ≥ 40% |
| **참여(Engagement)** | 세션당 미니게임/대결 참여 수, 평균 세션 길이 | 평균 세션 ≥ 6분, 세션당 상호작용 ≥ 3회 |
| **리텐션(Retention)** | D1 / D7 재방문율 | D1 ≥ 25%, D7 ≥ 10% (무로그인 기준 보정) |
| **바이럴(Viral)** | K-factor, 초대→입장 전환 | 초대 링크 클릭→입장 ≥ 30%, K ≥ 0.4 |
| **보존(Habit)** | 루팡왕 재도전율 | 전날 순위 확인 후 재접속 비율 ≥ 20% |

**측정상 유의점(무로그인 특성)**
- 유저 식별은 `session.ts`의 로컬 세션 ID 기반 → 브라우저/기기 교체 시 리텐션이 **과소 집계**된다. 절대값보다 **추세와 코호트 비교**를 신뢰한다.
- 봇/솔로 세션은 NSM에서 제외(2인+ 조건).

### 1.5 기능 로드맵 (우선순위)

> 라벨: **P0** = 런칭 필수(4주 내), **P1** = 성장(런칭 후 1~2개월), **P2** = 확장(3개월+). 현재 상태는 ✅ 완료 / 🟡 부분 / ⬜ 미착수.

#### (0) 현재 MVP — 완료 (Baseline)
실시간 급여 카운터(`SalaryEngine`), 생산성 지수(낮을수록 상위), 부장님 경보(`BossAlert`, 전원 위장 20초), 1:1 대결(`duel`, 전적), 미니게임 3종(`games`: 몰래딴짓/커피브레이크/가짜타이핑), 칭호·업적(`achievements`, common~legendary), 명예의 전당(`HallOfFame`, 통산 누적), 일별 루팡왕(`DailyKings`/`daily_records`), 이모지 반응(`Reactions`), 칼퇴 세리머니(컨페티), 초대 버튼(`InviteButton`), 솔로 모드(`/solo`).

### 1.6 리텐션 & 바이럴 루프 설계

**데일리 훅 (매일 돌아올 이유)**
1. **자정 리셋 + 루팡왕 박제:** 매일 0시 순위 확정, 전날 1위를 명예의 전당/일별 기록에 박제 → "오늘은 내가?" 재도전 동기.
2. **어제 결과 알림:** 재접속 시 "어제 우리 방 루팡왕은 OOO, 당신은 N위!" 즉시 노출.
3. **골든타임 넛지:** 선택 동의 시 오후 시간대(15시) 웹푸시로 "오늘의 루팡 콜".
4. **연속 참여 보상(가벼운 스트릭):** N일 연속 참여 칭호 — 단, 톤 유지("무의미한 성실함" 칭호처럼 자조적으로).

**리텐션 루프 도식**

```
접속 → 함께 딴짓/미니게임 → 자정 리셋으로 순위 박제
   ↑                                        │
   └──── "어제 결과 알림 + 재도전 욕구" ◀────┘
```

**바이럴 루프 (동료를 데려올 이유)**
- **핵심 마찰 제거:** 로그인·설치 0. 방코드/링크 한 번이면 즉시 합류 → 초대의 심리적 비용이 극히 낮음.
- **초대가 곧 게임 진행 조건:** 부장님 경보·1:1 대결·랭킹은 **사람이 많을수록 재밌다** → 시스템이 자연스럽게 "더 불러와"를 유도.
- **공유 가능한 밈 자산:** 결과 카드/루팡왕 박제 이미지를 카톡·슬랙·SNS에 공유 → 캡처 문화와 결합한 오가닉 확산.
- **OG 미리보기:** 공유 링크에 방 상태·루팡왕이 이미지로 보여 클릭률 상승.

**바이럴 루프 도식**

```
유저 → 결과 카드/방 링크 공유 → 동료 클릭(OG 미리보기)
  ↑                                     │
  └──── "많을수록 재밌음" 체감 후 재초대 ◀┘  → 신규 방 생성
```

**설계 주의:** 알림·스트릭은 **강요가 아닌 유머**로. "출석 강제" 느낌이 들면 톤이 죽고 이탈한다. 모든 리텐션 장치는 "아무 의미 없음" 자조 톤으로 감싼다.

### 1.7 수익화(Monetization) 전략

**대전제:** MVP는 무료. 수익화는 **톤을 해치지 않는 것**이 성공 조건이다. "딴짓하라고 부추기면서 돈을 뜯는다"는 인상은 브랜드 사망. → **"놀이의 확장에 돈을 쓴다"** 프레이밍만 허용.

**전환 가설 (Free → Paid)**
> "핵심 놀이는 영원히 무료. 유저는 *더 웃기게 꾸미거나(표현)*, *우리 팀만의 공간(소속)* 을 위해 지불한다."

**옵션별 트레이드오프**

| 모델 | 방식 | 장점 | 리스크/트레이드오프 | 우선순위 |
|---|---|---|---|---|
| **광고(Ads)** | 세션 사이 비침입 배너/네이티브 밈 광고 | 무료 유지, 대규모 트래픽 시 수익 | 톤 훼손 위험 大, 회사 PC 환경 부적합 | 낮음 (신중) |
| **커스터마이징 아이템** | 아바타·방 테마·칭호 스킨 판매(일회성/소액) | 톤 친화적, 표현 욕구와 직결, 죄책감 0 | 초기엔 콘텐츠 제작 리소스 필요 | **높음 (1순위)** |
| **구독(Pro)** | 상설방 다개설·상세 전적·시즌 특전·광고 제거 | 예측 가능한 반복 매출 | 무료로도 충분히 재밌어야 전환됨 | 중간 |
| **B2B(회사 단위)** | 팀빌딩 관리형 방, 워크샵/이벤트 패키지, 패러디 리포트 | 객단가 높음, 톤과 궁합("공식 딴짓 시간") | 세일즈 사이클 길고 컨셉 이해 필요 | 중간 (P2 검증) |

**단계적 로드맵**
1. **1단계(P1 검증):** 소액 커스터마이징(칭호 스킨/방 테마) — 지불의사·톤 수용성 테스트.
2. **2단계(P1~P2):** Pro 구독(상설방·시즌 특전) — 단골/조직방 유저 대상.
3. **3단계(P2):** B2B 팀빌딩 상품 파일럿 — "공식적으로 허가된 딴짓 시간"이라는 역설 세일즈.
4. 광고는 **최후 수단**, 도입 시에도 밈·유머 톤을 지키는 형태만.

**핵심 지표:** ARPPU보다 **"지불이 톤을 해쳤는가"**를 먼저 본다 → 유료 도입 코호트의 리텐션이 무료 대비 하락하면 즉시 롤백.

## 2. 개발 (Engineering)

### 2.1 현행 아키텍처 요약 & 강점

딴짓메이트는 Next.js 16 App Router · React 19 · TypeScript · Supabase(Postgres + Realtime) 기반의 실시간 멀티플레이어 웹앱이다. 인증 없이 닉네임 + 4자리 방코드로 입장하며, 방별 상태는 Supabase Postgres Changes 구독으로 전원에게 push된다. Railway(NIXPACKS)로 배포한다.

**잘 설계된 부분(실제 근거):**

- **순수 도메인 로직 분리 + 테스트.** 월급/카운트다운 계산이 `lib/salary.ts`에 전부 `now`를 명시적 인자로 받는 순수 함수로 추출되어 있고(`salary.ts:1-60`), 부수효과·전역시계가 없어 결정론적으로 테스트된다(`lib/salary.test.ts` 13개 케이스, `lib/achievements.test.ts` 9개 케이스). 이는 이 코드베이스의 가장 견고한 자산이다.
- **service role 키의 서버 격리.** `lib/supabaseServer.ts:5-16`은 `SUPABASE_SERVICE_ROLE_KEY`를 `NEXT_PUBLIC_` 접두사 없이 서버 런타임에서만 읽고, 브라우저 클라이언트(`lib/supabaseClient.ts`)는 anon 키만 사용한다. 키 분리 자체는 올바르게 되어 있다.
- **방코드 생성의 서버 게이트 + 충돌 재시도.** `rooms` INSERT는 RLS 정책이 없어 anon이 생성 불가하고, 오직 `app/api/room/route.ts:24-44`의 service role 경로로만 발급된다. 코드 중복(`23505`) 시 최대 8회 재시도(`route.ts:10,24`)로 방어한다.
- **혼동 문자 제거 방코드.** `lib/roomCode.ts:2`에서 O/0, I/1을 제외한 32자 알파벳을 사용해 공유·구두 전달 시 오독을 줄였다.
- **실시간 구조의 낙관적 UX.** 하트비트(`app/room/[code]/page.tsx:153-168`), 경보 낙관적 반영(`BossAlert.tsx:288-304`), 대결 브로드캐스트 상태머신(`DuelProvider.tsx`)이 잘 분리되어 있고, `replica identity full` + `eventsPerSecond: 10` 설정으로 방 단위 push가 동작한다.
- **경합 안전한 일일 리셋 선점.** `DailyReset.tsx:78-83`이 조건부 update(`.neq('last_reset_date', today)` + `.select()`)로 여러 클라이언트 중 한 명만 리셋을 확정하도록 낙관적 동시성 제어를 구현했다. 서버 크론 없이 클라이언트 선점으로 처리한 실용적 설계다.

**총평:** 프로토타입/MVP로서는 완성도가 높고 도메인 로직 품질이 좋다. 그러나 **신뢰 경계(trust boundary)가 사실상 없다**는 단일 근본 결함이 아래 대부분의 블로커를 파생시킨다.

### 2.2 기술 부채 & 런칭 블로커 (심각도 라벨)

### 2.3 런칭 전 필수(P0) 엔지니어링 작업

**P0-1. RLS 재설계 & 서버 권한 경계 (B1·B2·B3·B4 동시 해소) — 최우선.**
현재 "anon이 직접 DB에 write"하는 구조(`DailyReset`, `BossAlert`, `DuelProvider`, 하트비트)를 **모든 변경(mutation)을 서버 API 라우트 경유**로 전환한다. 방향:
- **읽기:** RLS SELECT를 방 단위로 좁힌다. 최소한 `rooms`/`players`/`room_state`/`daily_records`의 `using(true)`를 제거하고, 방코드를 아는 자만 자기 방을 읽도록 제한(예: 조회도 서버 라우트 경유로 방코드 검증 후 반환, 또는 RLS에 방 스코프 함수 적용).
- **쓰기:** anon UPDATE/INSERT 정책을 전부 제거하고, `getSupabaseServer()`(service role)만 write. 클라이언트는 `/api/room/heartbeat`, `/api/room/panic`, `/api/duel/result` 같은 라우트로 요청하고, 서버가 요청자 신원(§P0-2)과 방 소속·소유권을 검증한 뒤 반영한다. 이때 하트비트의 `loopang_sec`/`wins`는 클라이언트 값을 그대로 신뢰하지 말고 서버가 시간 델타·매치 존재를 기준으로 산출(B5·B7 해소, `wins`는 원자적 increment RPC).
- **트레이드오프:** write 지연이 직접 DB 대비 1홉 늘고, 낙관적 UX(예: `BossAlert.tsx:288`)를 유지하려면 클라이언트 낙관 반영 + 서버 확정 패턴이 필요하다. Realtime push는 그대로 유지 가능(서버 write도 postgres_changes를 발생시킴).

**P0-2. 최소 신원 체계 (B4).**
계정 로그인은 과하다. 대신 **입장 시 서버가 발급하는 방 스코프 토큰**(서명된 쿠키 또는 `player.id`+비밀 `player_secret`)을 도입해, 이후 모든 mutation 라우트가 "이 요청자가 이 player 본인인가"를 검증한다. `route.ts:83-93`의 join 응답에 secret을 포함하고 httpOnly 쿠키로 심는 방식이 localStorage보다 안전하다.
- **트레이드오프:** "로그인 없이 바로 시작"(`app/page.tsx:105`)이라는 제품 가치를 지키면서 최소 신원만 추가 — UX 저하 거의 없음.

**P0-3. 레이트리밋 / 어뷰징 방지 (B6·B8).**
방 생성·입장·경보·하트비트·대결 라우트에 IP+방 단위 레이트리밋을 건다. Railway 단일 인스턴스면 인메모리 토큰버킷으로 시작, 다중 인스턴스로 가면 Upstash Redis 등 외부 스토어로 이전. 방코드는 열거 방어를 위해 레이트리밋이 1차 방어선이며, 필요 시 코드 길이를 5~6자로 확대(공간 32^5≈3.36e7).

**P0-4. 관측성 최소셋 (B10).**
Sentry(클라이언트+서버 에러 추적) + 구조화 로깅(요청 ID·방코드·player 마스킹)을 API 라우트에 추가. 프라이버시 친화 애널리틱스(방 생성/입장/DAU/방 인원 분포) 1종. `console.error` 산재(`page.tsx:166` 등)를 로거로 통일.

**P0-5. 헬스체크 (B11).**
`app/api/health/route.ts` 추가 — Supabase 왕복 1회 + 200/503 반환. Railway 헬스체크 경로로 지정해 "떠 있지만 불통"을 감지.

**P0-6. CI 게이트 (B14).**
GitHub Actions에 `typecheck(tsc) → lint(eslint) → test(vitest run) → build(next build)`를 PR 필수 체크로. 지금 있는 22개 테스트가 배포 전 자동 실행되도록.

### 2.4 확장성(Scalability) 로드맵

**병목 1 — Supabase Realtime "Postgres Changes"의 구조적 한계 (B9).**
현재 `players` 하트비트(5초/인, `page.tsx:153-168`)가 방 크기 N에 대해 방당 약 **N²/5 메시지/초**를 만든다. Postgres Changes는 각 DB 변경을 **모든 구독에 대해 단일 스레드에서 RLS를 재평가**하는 것으로 알려진 병목이라, 방·인원이 늘면 지연/드롭이 먼저 터진다. 클라이언트당 5개 채널(`room:`, `room_state:`, `daily_reset:`, `duel:`, `reactions:`)을 여는 것도 커넥션 압박을 가중한다.
- **대안 A (권장, 저비용):** 현황·경보 같은 고빈도 방 상태를 **Postgres Changes → Broadcast 채널**로 이전. Broadcast는 DB를 거치지 않아(이미 `DuelProvider`·`Reactions`가 사용) fan-out이 훨씬 싸다. DB에는 저빈도 스냅샷(예: 30~60초, 이탈 감지용)만 영속화.
- **대안 B:** 하트비트 주기를 5초 → 10~15초로 늘리고 배칭. 채널을 방당 1개로 통합(이벤트 타입으로 분기).
- **대안 C (규모 확대 시):** presence는 Supabase Realtime Presence로, 상태 브로드캐스트는 전용 Realtime 서버/Ably 등으로 분리.
- **트레이드오프:** Broadcast로 옮기면 "영속 상태"가 약해져 새로고침 복원·감사(audit)가 어려워짐 → 스냅샷 영속화로 보완.

**병목 2 — DB 인덱싱/증가.**
`players_room_code_idx`(`schema.sql:25`)는 있으나, 명예의 전당/랭킹 정렬은 현재 클라이언트에서 수행(`page.tsx:27-31`, `HallOfFame`). 방이 커지거나 서버 집계로 옮기면 `(room_code, productivity, loopang_sec)` 복합 인덱스, `daily_records(room_code, date)`는 이미 PK. 하트비트 write 부하가 커지면 `players`의 hot-row 갱신에 대한 autovacuum/HOT update 튜닝 필요. `daily_records`는 시간이 지나면 파티셔닝(월 단위) 후보이나 P2 이전엔 불필요.

**병목 3 — 캐싱.**
방 제목·존재 여부(`page.tsx:81-94`, `route.ts:67-71`)는 변경이 드무니 짧은 TTL 캐시(엣지 또는 인메모리)로 DB 왕복을 줄일 수 있음.

**부하 테스트 계획.**
k6/Artillery로 (a) 방당 인원 5/20/50, (b) 동시 방 10/100/500 시나리오에서 Realtime 메시지 지연·드롭률·DB write TPS·Supabase 커넥션 상한을 측정. 목표 SLO 예: 상태 반영 p95 < 1.5s, 메시지 드롭 < 0.5%. P0 RLS/서버 경유 전환 후 재측정.

### 2.5 품질 / CI-CD

- **테스트 전략.** 현재 커버리지는 순수 로직 22개(`salary.test.ts` 13, `achievements.test.ts` 9)로 좋지만 **API 라우트·컴포넌트·통합 테스트가 전무**하다. 추가 우선순위: (1) `route.ts`의 create/join 계약 테스트(중복코드 재시도, 잘못된 action, 길이 검증), (2) P0에서 신설할 mutation 라우트의 인가(authz) 테스트 — "남의 행 수정 거부"를 회귀 방지 대상으로 고정, (3) `DailyReset` 롤오버 선점 로직의 동시성 테스트(순수 함수로 더 추출), (4) Playwright로 방 생성→입장→하트비트→경보 E2E 1개.
- **CI 파이프라인.** §P0-6 그대로. `vitest.config` 파일이 없으므로(확인됨) 커버리지 리포트·임계치를 위해 설정 파일 추가 권장.
- **프리뷰 환경.** Railway PR 환경 또는 브랜치별 프리뷰 + 별도 Supabase 프로젝트(스테이징)로 프로덕션 데이터 오염 방지. 현재 `.env.local` 단일 환경으로 보이며 스테이징 분리가 없음.
- **마이그레이션 관리 (B13).** 지금은 `schema.sql`을 SQL Editor에 수동 붙여넣기 + idempotent ALTER 누적 방식(`schema.sql:2,13,30-35,47-48`)이라 (a) 적용 여부·순서·환경 간 드리프트를 알 수 없고 (b) 롤백이 없다. **Supabase CLI 마이그레이션(`supabase/migrations/*.sql`, 타임스탬프 버전)**으로 전환하고 CI에서 `supabase db push`/`db diff`로 드리프트를 검출. 기존 idempotent 스크립트는 초기 baseline 마이그레이션으로 흡수.

- **백업/복구.** Supabase 자동 백업 활성화·주기 확인(무료 티어는 제한적 → Pro 이상 PITR 검토). 복구 리허설 1회. 데이터가 휘발성 오락 성격이므로 RPO/RTO는 느슨해도 무방하나, `daily_records`(통산 기록)는 사용자 애착 자산이라 보존 우선순위 상향.

### 3.1 포지셔닝 & 메시징

**포지셔닝 한 문장**
> 딴짓메이트는 "성실 경쟁"을 뒤집어, **덜 일할수록 이기는** 한국 직장인용 실시간 멀티플레이 놀이다. 게임이 아니라 **점심시간·티타임의 소셜 안주**.

**핵심 슬로건 후보**

| # | 슬로건 | 톤/용도 |
|---|--------|---------|
| A | **부장님 몰래, 다 같이 루팡하는 시간을 재보자** | 메인(현행 랜딩 카피 승계) — 안전빵, 설명형 |
| B | **일 안 할수록 1등** | 짧고 강한 훅 — 썸네일/광고 배너용 |
| C | **모든 지표는 아무 의미가 없습니다** | 브랜드 자학 캐치프레이즈 — 굿즈·엔딩·푸터용 |
| D | **오늘 우리 팀 루팡왕은 누구?** | 팀 소환·초대 트리거형 — 공유 문구용 |
| E | **월급은 흐르고, 나는 논다** | 감성/밈형 — 인스타 릴스 자막용 |

> 권장: 메인은 **A**, 퍼포먼스 훅은 **B**, 브랜드 시그니처는 **C**를 상시 병행.

**엘리베이터 피치 (15초)**
> "로그인 없이 닉네임이랑 4자리 코드만 넣으면, 우리 팀이 다 같이 '얼마나 농땡이 쳤는지'를 실시간으로 재는 놀이예요. 생산성이 낮을수록 랭킹이 올라가고, 부장님 뜨면 화면 전체가 엑셀로 위장되고, 옆자리랑 1:1 대결도 하고, 매일 '루팡왕'이 박제돼요. 진지한 지표는 하나도 없고 그냥 다 같이 웃자고 만든 거예요."

**브랜드 톤 가이드**

| 항목 | Do | Don't |
|------|----|----|
| 화법 | 병맛·자학·직장인 밈("월급은 통장을 스칠 뿐") | 진지한 생산성 훈계, 자기계발 톤 |
| 호칭 | "루팡러", "부장님", "우리 팀" | "사용자", "고객님" 같은 딱딱한 서비스어 |
| 이모지 | 🕵️ 💀 ☕ 🏖️ 적극 사용 (제품 칭호와 일관) | 남발해 가독성 해치기 |
| 태도 | "이거 해봤자 아무 의미 없음 ㅋㅋ" 를 당당하게 | 실제 근무태만을 조장/미화하는 발언 |
| 안전선 | "점심시간·티타임에 잠깐" 프레이밍 유지 | "일하지 말자"는 직접적 선동 |

### 3.5 바이럴 / 그로스 루프

**핵심 루프**
```
① 한 명이 방 생성 → ② 팀 단톡에 4자리 코드 초대 → ③ 여러 명 입장(방 확산)
        ↑                                                      ↓
⑥ 다음날 재초대·새 방 ← ⑤ 루팡왕 박제 카드 밖으로 공유 ← ④ 함께 놀며 루팡왕/칭호 획득
```

이 루프에서 **K-factor = (한 명이 보내는 초대 수) × (초대 수락률)**. 무가입·4자리 코드 구조라 수락률이 이미 높음 → 관건은 **"공유 트리거를 얼마나 많이, 자랑스럽게" 심느냐**.

**마케팅 관점에서 필요한 제품 내 공유 기능 (제품팀에 요청)**

| 우선순위 | 공유 트리거 | 왜 필요한가 (마케팅 논리) | 상태 |
|----------|-------------|---------------------------|------|
| P0 | **초대 버튼 → 카톡/링크 원터치 공유** | 방 확산의 심장. 코드 복붙 마찰 제거 | 존재(InviteButton) — 카톡 공유 문구 최적화 필요 |
| P0 | **루팡왕 박제 → 공유용 이미지 카드 자동 생성** | 가장 자랑스러운 순간을 밖으로. UGC 소스 | 신규 요청 |
| P0 | **칭호/업적 획득 → "내 칭호 자랑하기" 이미지 다운로드/공유** | 챌린지 밈의 연료(💀완전체 루팡 등) | 신규 요청 |

## 4. 앱 전환 (Mobile App)

### 4.1 왜/언제 앱이 필요한가

먼저 솔직한 판단부터. **딴짓메이트의 핵심 플레이 루프는 본질적으로 데스크톱(업무용 PC) 경험**이다. "부장님 경보 → 위장 화면"의 위장 화면 3종(스프레드시트·메일·코드, `components/BossAlert.tsx:55-197`)은 회사 모니터에서 상사 시야를 속이기 위한 장치다. 스마트폰에서는 애초에 화면을 잠그거나 엎으면 그만이라 위장 화면의 존재 이유가 약하다. 즉 **모바일 앱이 데스크톱 플레이 표면을 복제하는 것은 목표가 아니다.**

그렇다면 앱은 무엇을 위한 채널인가. 웹만으로 부족하고 앱(또는 설치형 웹)이 실질 가치를 주는 지점은 명확히 구분된다.

| 축 | 순수 웹(현재)으로 충분 | 앱/설치형이 주는 실질 가치 |
|----|----------------------|--------------------------|
| 플레이(딴짓·위장·카운터) | ✅ 데스크톱 브라우저가 최적 | 추가 가치 낮음 |
| 실시간 대결·리액션 | ✅ Supabase Realtime로 충분 | 추가 가치 낮음 |
| **재참여(re-engagement)** | ❌ 탭 닫으면 끝 | ⭐ 푸시로 "루팡왕 발표/칼퇴/도전장" 재소환 |
| **부재 중 이벤트 수신** | ❌ 연결된 클라이언트만 수신 | ⭐ 백그라운드 푸시로 도달 |
| **홈스크린 상주·존재감** | ❌ URL 재입력 필요 | ⭐ 아이콘 1탭 진입 |
| **바이럴·초대** | 🔸 `navigator.share` 이미 사용(`components/InviteButton.tsx:14-16`) | ⭐ 딥링크로 방까지 원탭 진입 |
| **스토어 노출·신뢰도** | ❌ 없음 | 🔸 검색·마케팅 채널(리젝 리스크 있음, §4.6) |

**결론:** 앱은 "플레이어"가 아니라 **컴패니언(알림·바이럴·재참여) 표면**이다. 이 정의가 아래 모든 선택을 좌우한다 — 무거운 네이티브 UI 재작성은 과잉이고, 알림·딥링크·홈스크린 상주를 값싸게 얻는 경로가 정답이다.

> **⚠️ 최우선 아키텍처 사실 (셸 선택보다 중요):** 현재 앱에는 **서버 사이드 스케줄러/트리거가 없다.** 일일 리셋·루팡왕 박제는 자정에 "마침 접속해 있던 클라이언트"가 처리하고(`components/DailyReset.tsx:74-107`), 칼퇴는 클라이언트에서 `workPhase()`로 계산하며(`lib/salary.ts:49-60`), 부장님 경보·대결 도전은 Realtime 채널에 **현재 구독 중인** 클라이언트에게만 도달한다(`components/BossAlert.tsx:307`, `components/duel/DuelProvider.tsx:147`). 따라서 **앱을 닫았거나 백그라운드일 때 울리는 푸시는 어떤 셸(PWA/Capacitor/RN)을 골라도 지금 구조로는 불가능하다.** 푸시의 선행 조건은 셸이 아니라 백엔드다(§4.3 Phase 0).

### 4.2 전환 옵션 비교 (핵심)

| 옵션 | 개발비용 | 코드 재사용성 | 성능 | 네이티브 기능(푸시) | 스토어 정책 | 유지보수 | 이 프로젝트 적합도 |
|------|---------|--------------|------|-------------------|------------|---------|------------------|
| **(A) PWA** (설치형 웹 + Web Push) | **매우 낮음** — manifest + 서비스워커 + 아이콘 추가, 기존 Next.js 그대로 | **100%** — 코드 무수정 | 웹과 동일(이 앱은 가벼워 충분) | Android·데스크톱 Chrome/Edge Web Push ✅ / **iOS는 16.4+ & 홈화면 설치 시에만** 🔸 | 스토어 미등록(설치 배너로 배포), 심사 없음 | **최저**(단일 코드베이스) | ⭐ **1순위 (런칭)** |
| **(B) Capacitor 래핑** | 낮음~중간 — 웹을 네이티브 셸에 로드 + 플러그인 배선 | **~95%** — UI 그대로, 브리지만 추가 | 웹뷰(이 앱 수준엔 충분) | FCM/APNs 네이티브 푸시로 **iOS 포함 완전 지원** ✅ | 등록 가능. 단 "얇은 웹래퍼" 리젝 리스크(Apple 4.2, §4.6) | 웹 코드 공유 + 네이티브 빌드 파이프라인 추가 | ⭐ **2순위 (iOS 푸시·스토어 필요 시)** |
| **(C) React Native/Expo 신규** | **높음** — UI 전면 재작성 | **~30~40%** — 순수 로직 `lib/*`만, 컴포넌트 전량 재작성 | 네이티브 최상 | 완전 지원 ✅ | 실질 네이티브 → 리젝 리스크 최소 | **최고(이중 코드베이스)** | ▲ 과잉 (앱이 주력 채널·네이티브 UX가 차별점일 때만) |
| **(D) 하이브리드**(웹 우선 + 얇은 네이티브 셸) | = (B)의 전략적 표현 | ~95% | 웹뷰 | 완전 지원 ✅ | = (B) | = (B) | ⭐ (B)와 통합 추천 |

**해설**

- **(A) PWA**는 배포 기반(Railway Node 서버, `next start`)에서 서비스워커와 Web Push 구독 엔드포인트(VAPID)를 **같은 Next.js 앱**의 route handler(`app/api/push/`)로 그대로 서빙할 수 있어 인프라 추가가 사실상 없다. 유일한 약점은 iOS Web Push 제약(홈화면 설치 필수, Safari 탭 푸시 불가).
- **(B)/(D)는 사실상 하나의 전략**이다. 웹을 정본(canonical)으로 두고 Capacitor 셸에는 **푸시·딥링크·공유** 같은 네이티브 능력만 얇게 얹는다. iOS 푸시의 신뢰도와 스토어 노출이 필요해지는 순간의 정답.
- **(C)는 현 단계에서 명백한 과잉**이다. 재사용 가능한 것은 순수 로직뿐(§4.4)이고, DOM 기반 컴포넌트(위장 화면·PlayerCard·게임들)를 전부 다시 짜야 하며 웹/앱 2개 코드베이스를 영구 유지해야 한다. §4.1의 "앱=컴패니언" 정의와 정면충돌한다.

**추천안: (A) PWA로 런칭 → 필요가 증명되면 (B/D) Capacitor 얇은 셸. (C)는 지표가 강하게 정당화할 때만.**

**핵심:** Phase 0가 진짜 관문이다. 이걸 하면 **PWA만으로도** Android/데스크톱 사용자에게 "루팡왕 발표", "곧 칼퇴", "도전장 도착" 푸시가 나간다. iOS 프리미엄 푸시가 필요할 때 비로소 Phase 2.

### 4.4 재사용 vs 재작성

현재 코드 자산을 셸별로 분류. (A)/(B)는 웹 코드를 그대로 실행하므로 재사용 이슈가 거의 없고, **분류가 중요한 건 (C) RN 시나리오**다.

### 4.7 의사결정 권고

**① 런칭에 앱은 필수 아니다 — PWA로 시작한다.** 핵심 플레이는 데스크톱 웹이 최적이고(§4.1), 스토어 앱은 데스크톱 컨셉을 복제하지 않으며 컴패니언 역할일 뿐이다. 네이티브 스토어 앱을 런칭 게이트에 넣으면 심사 리스크(§4.6)와 일정만 늘고 핵심 가치는 커지지 않는다.

**② 진짜 우선순위는 셸이 아니라 백엔드 트리거(Phase 0)다.** 서버 사이드 스케줄러/이벤트 팬아웃이 없으면 어떤 셸을 골라도 "닫힌 앱을 다시 부르는" 재참여 푸시가 불가능하다(§4.1 경고). 이건 **PWA에도 똑같이 필요**하므로, 앱 여부 결정과 무관하게 가장 먼저 착수한다.

**③ 지표를 보고 (B) Capacitor로 승격한다.** 다음 신호 중 2개 이상이면 Phase 2 착수:
- Android/데스크톱 Web Push 오픈율이 재참여를 유의미하게 끌어올림(→ iOS까지 확대 가치 입증)
- iOS 사용자 비중이 높고 홈화면 설치 전환율이 낮음(iOS Web Push 제약이 병목)
- 스토어 검색·오가닉 유입이 신규 성장의 필요 채널로 확인됨

**④ (C) React Native는 당분간 배제한다.** 재사용률이 순수 로직에 그치고(§4.4) 이중 코드베이스 부담이 큰데, "앱=컴패니언" 정의에서 네이티브 UI의 차별 가치가 아직 없다. 앱이 주력 채널이 되고 (B)의 웹뷰로 UX 한계에 부딪힐 때만 재평가.

**한 줄 권고: "PWA로 런칭하되, 그 전에 서버 사이드 알림 트리거(Phase 0)부터 만든다. iOS 프리미엄 푸시·스토어 노출이 지표로 정당화되면 Capacitor 얇은 셸로 승격한다."**

## 부록 A. 이 PRD의 작성 방식

이 문서는 4개 도메인(기획·개발·마케팅·앱 전환)을 **병렬 전문 에이전트 팀**이 각각 분석·작성하고, 그 결과를 하나로 통합·중복 제거하여 만들었다. 개발·앱 전환 섹션은 실제 코드베이스를 `file:line` 단위로 근거를 달아 진단했다. 각 도메인의 상세 근거·표는 본문 §1~§4를 참조한다.
